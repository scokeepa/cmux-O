## 레벨 아이콘과 별명

| 레벨 | 영문명 | 한국어 별명 | 아이콘 | 서사 단계 |
|------|--------|------------|--------|----------|
| L1 | Dreamer | 꿈꾸는 사람 | ○ | **수동** |
| L2 | Sketcher | 스케치하는 사람 | ◐ | 수동→능동 |
| L3 | Shaper | 다듬는 사람 | ● | **능동** |
| L4 | Planner | 설계자 | ◆ | 능동→주도 |
| L5 | Strategist | 전략가 | ⬡ | **주도** |
| L6 | Director | 감독 | ✦ | 주도→창조 |

## 6대 기술 차원 정의

바선생은 AI 활용 기술을 6개 독립 축으로 측정한다. 테니스 NTRP가 포핸드/백핸드/서브/발리/풋워크/전략을 각각 평가하듯, 각 축을 따로 보고 종합한다.

| 코드 | 기술 차원 | Designer에서의 의미 |
|------|----------|---------------------|
| **DECOMP** | 작업 분해 | 큰 기획을 단계별로 쪼개는 능력. "콘텐츠 기획해줘"를 문제정의→페르소나→전략→실행계획으로 분해하는 것. |
| **VERIFY** | 검증 전략 | AI가 만든 기획서를 비판적으로 검토하고, 약점을 찾고, 실현 가능성을 확인하는 능력. |
| **ORCH** | 오케스트레이션 | 프레임워크, 템플릿, 분석 도구를 조합하여 기획 워크플로우를 구성하는 능력. |
| **FAIL** | 실패 대응 | 기획의 약점, 잘못된 가정, 실현 불가능한 전략에 대응하는 능력. |
| **CTX** | 맥락 관리 | 타깃 사용자, 비즈니스 목표, 제약 조건, 이해관계자 정보를 AI에게 제공하는 능력. Designer의 핵심 축. |
| **META** | 메타인지 | "나는 어떤 관점에서 기획하고 있는가?"를 인식하고, 시스템적 사고를 적용하는 능력. Designer의 핵심 축. |

## Designer 가중치

Designer는 **기획하는 사람**이다. 맥락(CTX)과 메타인지(META)가 합쳐서 성장의 45%를 차지한다. 좋은 기획은 맥락 이해에서 시작하고, 시스템적 사고로 완성된다.

| 기술 차원 | 가중치 | 이유 |
|----------|--------|------|
| **DECOMP** | 20% | 기획을 구조화하고 단계별로 나누는 것이 중요 |
| **VERIFY** | 15% | 기획의 약점을 찾고 검증하는 것도 필요 |
| **ORCH** | 15% | 프레임워크와 도구 활용 능력 |
| **FAIL** | 15% | 잘못된 가정에 대응하는 능력 |
| **CTX** | **15%** | 타깃, 제약, 이해관계자 정보를 정확히 전달하는 것이 기획의 기초 |
| **META** | **20%** | 시스템적 사고, 2차 효과 예측, 전략적 의사결정이 Designer의 궁극적 성장 목표 |

FAIL   ████████████████████            15%

## 7단계 레벨 정의

### Level 1 (L1.0) — Dreamer (꿈꾸는 사람) ○

**NTRP 대응**: 1.5 (공을 코트 안에 넣는 것이 목표)
**핵심 특성**: 요청이 막연하고 구체성이 없다. "좋은 거 만들어줘", "트렌디하게 해줘" 수준. AI 결과물을 그대로 최종본으로 사용한다. 타깃이 "모든 사람"이거나 정의되지 않는다.
**판별 질문**: "AI에게 기획을 요청할 때, 원하는 결과를 얻지 못하면 어떻게 하나요?" → "다시 물어본다" / "포기한다"
**인지 수준**: Remember (기억) — 나열하다, 인식하다, 따라하다

### Level 1.5 (L1.5) — Dreamer+ ○

**전환 중 행동**: 가끔 목적을 언급하기 시작한다. "~하려고 하는데 기획해줘". 하지만 구체적 타깃이나 제약은 아직 없다.

### Level 2 (L2.0) — Sketcher (스케치하는 사람) ◐

**NTRP 대응**: 2.5 (느린 랠리 지속 가능)
**핵심 특성**: 목적과 대상을 명시할 수 있다. 제약 조건을 일부 포함한다 (예: "2주 안에", "예산 없이"). AI 결과물에 간단한 피드백을 줄 수 있다. 핵심 가치보다 기능 목록에 집중하는 경향.
**판별 질문**: "기획 요청에 목적과 대상이 포함되나요?" → "가끔 포함한다"
**인지 수준**: Understand (이해) — 설명하다, 비교하다, 질문하다

### Level 2.5 (L2.5) — Sketcher+ ◐

**전환 중 행동**: 프레임워크 개념을 접하기 시작한다. "페르소나를 만들어봐"라고 요청하지만, 직접 설계하지는 않고 AI에게 맡긴다.

### Level 3 (L3.0) — Shaper (다듬는 사람) ●

**NTRP 대응**: 3.0 (중간 속도 랠리 안정적)
**핵심 특성**: 구조화된 기획을 작성할 수 있다 (문제→솔루션→실행). 페르소나를 활용하고 사용자 관점에서 생각한다. 프레임워크를 도구로 활용한다 (BMC, 여정 맵 등). 여러 번 피드백 루프를 돌린다.
**판별 질문**: "기획 요청에 프레임워크나 구조를 직접 설계하나요?" → "대부분 그렇다"
**인지 수준**: Apply (적용) — 적용하다, 실행하다, 구현하다

### Level 3.5 (L3.5) — Shaper+ ●

**전환 중 행동**: 대안을 비교하려는 시도가 나타나지만 체계적이지 않다. "다른 방향도 있을까?"를 묻기 시작한다. 가설을 의식하기 시작하지만 아직 명시적으로 기술하지 않는다.

### Level 4 (L4.0) — Planner (설계자) ◆

**NTRP 대응**: 4.0 (포핸드, 백핸드 모두 방향+깊이 제어)
**핵심 특성**: 대안을 비교하고 선택 근거를 설명할 수 있다. 기획에 검증 가능한 가설이 포함된다. 실현 가능성을 제약 조건과 함께 검토한다. 이해관계자 관점을 포함한다. OKR/KPI 수준의 성과 지표를 정의한다.
**판별 질문**: "기획에서 대안을 비교하고 선택 근거를 제시하나요?" → "대부분 그렇다"
**인지 수준**: Analyze (분석) — 분석하다, 구별하다, 비판하다

### Level 4.5 (L4.5) — Planner+ ◆

**전환 중 행동**: 시스템적 사고의 조짐이 보인다. "이 전략이 다른 영역에 미치는 영향은?"을 가끔 묻는다. 2차 효과를 인식하기 시작하지만 체계적이지 않다.

### Level 5 (L5.0) — Strategist (전략가) ⬡

**NTRP 대응**: 5.0 (예측 능력 발달, 고급 샷 구사)
**핵심 특성**: 전체 시스템을 보고 기획한다 (2차, 3차 효과 예측). 단기 실행과 장기 전략을 연결한다. 기획이 비즈니스 목표에 정렬되어 있다. AI를 사고 파트너로 활용한다 (검증, 비판, 탐색). 기획 과정 자체를 지속적으로 개선한다.
**판별 질문**: "기획의 2차, 3차 효과를 예측하나요?" → "체계적으로 검토한다"
**인지 수준**: Evaluate (평가) — 평가하다, 비교하다, 판단하다

> **L4→L5는 "80%의 벽" 돌파 지점이다.** "기획서를 만드는 사람"(Planner)에서 "전략을 설계하는 사람"(Strategist)으로 전환하는 순간.

### Level 5.5 (L5.5) — Strategist+ ⬡

**전환 중 행동**: 기획 프로세스 자체를 개선하려는 시도가 보인다. 기획 결과를 다른 사람에게 교육하거나 공유한다. 여러 프로젝트에 걸친 전략적 패턴을 인식한다.

### Level 6 (L6.0) — Director (감독) ✦

**NTRP 대응**: 6.0 (세미프로, 전국 수준)
**핵심 특성**: AI 기반 기획 시스템을 설계하고 운영한다. 조직/팀 수준의 전략 프레임워크를 구축한다. 멀티에이전트를 활용한 종합 기획을 수행한다.
**판별 질문**: "AI를 활용한 반복 가능한 기획 프로세스를 만든 적 있나요?" → "설계하고 운영 중이다"
**인지 수준**: Create (창조-응용) — 설계하다, 구성하다, 통합하다

### Level 6.5 (L6.5) — Director+ ✦

**전환 중 행동**: 자신만의 기획 방법론을 정립하기 시작한다. 커뮤니티나 팀에서 전략 프레임워크를 공유한다. AI 도구의 한계를 극복하는 커스텀 기획 시스템을 개발한다.

### Level 7 (L7.0) — Visionary (비전가) ◈

**NTRP 대응**: 7.0 (세계 최상위)
**핵심 특성**: AI 활용 기획의 새로운 방법론이나 패러다임을 창조한다. 커뮤니티/산업에 기여한다. 전략적 의사결정의 새로운 프레임워크를 제시한다.
**판별 질문**: "다른 사람들이 참고하는 기획 방법론이나 프레임워크를 만든 적 있나요?" → "만들었고, 다른 사람들이 사용하고 있다"
**인지 수준**: Create (창조-혁신) — 창조하다, 발명하다, 혁신하다

## 6축 x 7레벨 행동 기술자 매트릭스

각 칸에는 **관찰 가능한 행동**(~를 한다)과 **자동 감지 신호**를 함께 기술한다. 0.5 레벨은 전환 중 행동이다.

### DECOMP (작업 분해) — Designer의 기획 구조화

> 큰 기획 과제를 단계별로 나누고, 각 단계의 산출물을 정의하는 행동

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | 하나의 큰 요청을 통째로 던진다. "콘텐츠 기획해줘" 수준. | avg_msg_len < 100자, 번호 목록 비율 = 0% |
| **L1.5** | 가끔 "먼저 아이디어부터"라는 표현이 등장하지만 체계적이지 않다. | avg_msg_len 100~200자, 순서 표현 비율 < 5% |
| **L2.0** | 요청에 여러 항목을 나열한다. "제목도 정해주고 내용도 짜줘". 순서는 고려하지 않는다. | avg_msg_len 150~300자, 나열형 요청 비율 > 10% |
| **L2.5** | "먼저 타깃을 정하고, 그 다음 콘텐츠를 기획해줘" 형태의 순서 지정이 나타난다. | 순서 접속사 포함 비율 > 15% |
| **L3.0** | 기획을 문제정의→솔루션→실행의 3단계 이상으로 분해한다. 각 단계에 산출물을 기대한다. | 번호 목록 포함 비율 > 20%, 단계 수 평균 3+ |
| **L3.5** | 단계 간 의존관계를 인식한다. "페르소나가 정해져야 콘텐츠 방향을 잡을 수 있다". | 조건 표현 포함 비율 > 10% |
| **L4.0** | 기획을 전략-전술-실행의 계층으로 분해한다. 각 계층의 입력/출력을 명시한다. | 계층적 분해 패턴, 분해 단계 4+ |
| **L4.5** | 가설 검증 단계를 먼저 거친 후 본 기획을 진행하는 패턴이 나타난다. | 탐색형 후 기획 패턴 > 20% |
| **L5.0** | 전체 프로젝트를 분기/월 단위로 설계한다. 각 단계의 KPI와 마일스톤을 정의한다. | 프로젝트 구조 언급 > 30%, 다중 세션 연속 |
| **L5.5** | 각 기획 단계를 다른 에이전트/도구에 배분한다. 리서치 → 기획 → 검증의 파이프라인. | has_orchestration = True, orch_tool_count > 3 |
| **L6.0** | 멀티에이전트 기획 팀을 구성한다. 리서처, 기획자, 비평가 역할 배분. | teamName 존재, orch_tool_count > 10 |
| **L6.5** | 반복 가능한 기획 템플릿과 워크플로우를 만들어 조직에 배포한다. | 커스텀 명령어/스킬 사용 |
| **L7.0** | 새로운 기획 분해 방법론을 창안하고 체계화한다. | 외부 공유 증거 |

### VERIFY (검증 전략) — Designer의 기획 검증

> AI가 만든 기획서의 약점을 찾고, 실현 가능성을 확인하는 행동

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | AI 기획 결과를 그대로 수용한다. "좋아, 이걸로 하자". | verification_ratio = 0 |
| **L1.5** | 결과가 마음에 안 들면 "다시 해줘"라고 하지만, 무엇이 부족한지 구체적으로 말하지 않는다. | correction_ratio < 0.05 |
| **L2.0** | "이거 괜찮아?"라고 확인 질문을 한다. AI의 "괜찮다"에 안심한다. | verification_ratio > 0, followup_ratio > 0.03 |
| **L2.5** | 특정 부분을 지적한다. "이 부분이 약한 것 같은데..." 하지만 대안을 제시하지는 않는다. | correction_ratio > 0.05 |
| **L3.0** | 기획의 논리적 연결성을 확인한다. "문제정의와 솔루션이 연결되나?" | verification_ratio > 0.15, 논리 검증 키워드 |
| **L3.5** | "이 기획의 약점을 찾아줘", "리스크가 뭐야?"를 요청하기 시작한다. | "약점"/"리스크"/"risk" 키워드 > 5% |
| **L4.0** | 기획의 가정을 확인한다. "이 가정이 틀렸다면?", "대안은?" 실현 가능성을 제약 조건과 함께 검토한다. | verification_ratio > 0.2, correction_ratio > 0.1 |
| **L4.5** | 이해관계자별 관점에서 기획을 재검증한다. "사용자 입장에서는?", "투자자 관점에서는?" | alternative_ratio > 0.05, 이해관계자 키워드 |
| **L5.0** | 기획 품질 기준을 사전에 정의한다. "이 기준을 만족하는지 확인해". 검증 체크리스트를 사용한다. | constraint_ratio > 0.1, 기준 키워드 |
| **L5.5** | 기획 검증을 체계화한다. 가설 검증, 사용자 테스트, 시장 조사를 통합한다. | 자동화 도구 사용, 체계적 키워드 |
| **L6.0** | 기획 검증 시스템을 설계한다. 팀 수준의 기획 리뷰 프로세스를 구축한다. | 시스템 수준 검증 키워드 |
| **L6.5** | 기획 품질 메트릭을 추적하고 트렌드를 분석한다. | 메트릭 추적 패턴 |
| **L7.0** | 새로운 기획 검증 방법론이나 품질 프레임워크를 창안한다. | 외부 공유 증거 |

### ORCH (오케스트레이션) — Designer의 도구 활용

> 프레임워크, 템플릿, 분석 도구를 조합하여 기획 워크플로우를 구성하는 행동

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | 단일 채팅에서 텍스트 대화만 한다. | tool_diversity <= 2 |
| **L1.5** | 참고 자료(이미지, 링크)를 가끔 첨부한다. | tool_diversity = 2~3 |
| **L2.0** | 2~3개 도구를 순차적으로 사용한다. AI + 검색 + 문서 편집. | tool_diversity = 3~4 |
| **L2.5** | 프레임워크 템플릿(BMC, 페르소나 시트 등)을 참고하기 시작한다. | tool_diversity = 4, 프레임워크 키워드 |
| **L3.0** | 4~6개 도구를 목적에 맞게 선택한다. 리서치 + 분석 + 기획서 작성의 기본 체인. | tool_diversity = 4~6, tools_per_turn > 3 |
| **L3.5** | 프레임워크를 상황에 맞게 선택한다. "이건 SWOT보다 여정 맵이 맞을 것 같아". | 프레임워크 선택 이유 언급, tool_diversity > 5 |
| **L4.0** | 7~9개 도구를 조합한다. 경쟁 분석 + 사용자 조사 + 전략 수립 + 실행 계획의 통합 워크플로우. | tool_diversity = 7~9, tools_per_turn > 5 |
| **L4.5** | 자주 쓰는 기획 도구 조합을 패턴화한다. 반복 가능한 기획 프로세스가 형성된다. | 반복 도구 패턴, tool_diversity > 8 |
| **L5.0** | 멀티에이전트를 활용하여 기획의 각 단계를 병렬 처리한다. | has_orchestration = True, orch_tool_count > 5 |
| **L5.5** | 에이전트 간 역할 분담을 설계한다. 리서치 에이전트, 기획 에이전트, 비판 에이전트. | orch_tool_count > 8 |
| **L6.0** | 멀티에이전트 기획 시스템을 운영한다. 전략 수립부터 실행 계획까지 통합 자동화. | teamName 존재, orch_tool_count > 15 |
| **L6.5** | 커스텀 MCP, 데이터 분석 파이프라인을 통합한 기획 시스템을 구축한다. | mcp_tool_count > 5 |
| **L7.0** | 새로운 기획 오케스트레이션 패턴이나 프레임워크를 창안한다. | tool_diversity > 15, 외부 기여 증거 |

### FAIL (실패 대응) — Designer의 가정 검증

> 기획의 약점, 잘못된 가정, 실현 불가능한 전략에 대응하는 행동

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | 기획이 안 되면 "다시 해줘"만 요청한다. 무엇이 잘못됐는지 분석하지 않는다. | 동일 메시지 반복 비율 > 30% |
| **L1.5** | "이거 아닌 것 같은데..."라고 느끼지만 구체적으로 뭐가 문제인지 말하지 못한다. | 불만족 표현 있으나 구체적 지적 없음 |
| **L2.0** | 기획의 문제점을 대략적으로 인식한다. "이 부분이 좀 현실적이지 않은데..." | "현실"/"가능"/"실현" 패턴, followup_ratio > 0.03 |
| **L2.5** | 구체적 약점을 지적한다. "이 예산으로는 이 규모가 안 될 것 같아". | 제약/한계 지적 패턴 |
| **L3.0** | 기획의 가정을 인식하고 검토한다. "이건 타깃이 이렇게 행동한다는 전제인데, 맞을까?" | 가정 검토 패턴 > 10% |
| **L3.5** | 실패한 기획에서 교훈을 추출한다. "지난번 기획이 안 된 이유가 X였으니까 이번엔..." | 과거 실패 참조 패턴 > 5% |
| **L4.0** | 기획의 리스크를 사전에 식별하고 대응 방안을 마련한다. "이 전략의 리스크 3가지와 대응 방안은?" | alternative_ratio > 0.05, "리스크" 키워드 |
| **L4.5** | 기획의 가정이 틀렸을 때의 대안(Plan B)을 미리 준비한다. | 대안 준비 패턴, "Plan B" 키워드 |
| **L5.0** | 전략적 실패 분석을 수행한다. 가설 검증 결과를 바탕으로 전략을 피벗한다. | 체계적 분석 키워드, recovery 성공률 > 70% |
| **L5.5** | 기획 실패 패턴을 데이터로 추적한다. 어떤 유형의 가정이 자주 틀리는지 분석한다. | 데이터 분석 패턴 |
| **L6.0** | 팀/조직 수준의 기획 실패 대응 프로세스를 설계한다. 피벗 기준, 중단 기준을 체계화한다. | 시스템 수준 프로세스 키워드 |
| **L6.5** | 기획 실패 데이터를 수집하고 조직 학습에 활용한다. 사후 분석(post-mortem) 문화를 구축한다. | 분석/학습 패턴 |
| **L7.0** | 새로운 전략적 실패 대응 방법론이나 기획 탄력성 프레임워크를 창안한다. | 외부 기여 증거 |

### CTX (맥락 관리) — Designer의 이해관계자 이해

> 타깃 사용자, 비즈니스 목표, 제약 조건, 이해관계자를 AI에게 제공하는 행동. Designer의 핵심 축.

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | 맥락 없이 요청한다. "기획해줘". 타깃, 목적, 제약 전혀 없음. | context_specificity_ratio < 0.3 |
| **L1.5** | 가끔 목적을 언급한다. "인스타 좋아요 늘리려고 하는데". | 목적 표현 비율 < 10% |
| **L2.0** | 목적과 대상을 명시한다. "30대 직장인 대상 뷰티 콘텐츠". 제약도 일부 포함. | context_specificity_ratio > 0.4 |
| **L2.5** | 기술 스택이나 플랫폼 제약을 언급한다. "인스타그램 릴스", "예산 50만 원". | context_specificity_ratio > 0.5 |
| **L3.0** | 페르소나를 제공하거나 요청한다. 사용자 여정의 특정 단계를 명시한다. | context_specificity_ratio > 0.6, 페르소나 키워드 |
| **L3.5** | 비즈니스 목표와 기획의 연결을 명시한다. "이 캠페인의 비즈니스 목표는 리텐션 향상". | constraint_ratio > 0.03, 비즈니스 키워드 |
| **L4.0** | 이해관계자별 니즈를 구분하여 제공한다. 성공 지표(KPI)를 명시한다. | constraint_ratio > 0.05, 이해관계자/KPI 키워드 |
| **L4.5** | 기획의 상위 전략과 하위 실행의 맥락을 모두 제공한다. "전사 전략은 X이고, 이 기획은 그 중 Y에 해당해". | 전략 맥락 표현, avg_msg_len > 600 |
| **L5.0** | 시장 환경, 경쟁 현황, 내부 역량을 종합적으로 제공한다. 맥락의 우선순위를 정한다. | 종합 맥락 키워드, 맥락 파일 참조 |
| **L5.5** | 맥락 제공을 자동화한다. 브랜드 가이드라인, 타깃 프로필, 경쟁 분석을 사전 준비한다. | 설정 파일/템플릿 사용 |
| **L6.0** | 조직 수준의 전략 맥락 관리 체계를 구축한다. 공유 전략 문서, 브랜드 아키텍처를 설계한다. | 구조화된 맥락 파일 다수 |
| **L6.5** | 맥락 품질을 최적화한다. 어떤 맥락 정보가 기획 품질에 가장 큰 영향을 미치는지 분석한다. | 맥락 최적화/A/B 비교 |
| **L7.0** | 새로운 전략 맥락 관리 방법론을 창안하고 공유한다. | 외부 기여 증거 |

### META (메타인지) — Designer의 시스템적 사고

> "나는 어떤 관점에서 기획하고 있는가?"를 인식하고 전략적으로 조정하는 행동. Designer의 핵심 축.

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | 기획 방식에 대한 성찰 없음. "그냥 기획해본다". | metacognitive_ratio = 0 |
| **L1.5** | "AI가 기획을 잘 해주네" / "좀 부족한데"라는 수준의 인상적 평가. | 평가 표현 있으나 기준 없음 |
| **L2.0** | AI가 잘하는 기획과 못하는 기획을 경험적으로 느낀다. | 경험적 평가 패턴 |
| **L2.5** | 요청 방식이 기획 결과에 영향을 미친다는 것을 인식한다. | 요청 방식 언급 패턴 |
| **L3.0** | 프레임워크를 의식적으로 선택한다. "이 상황에는 SWOT보다 5 Forces가 적합하다". | 프레임워크 선택 키워드 |
| **L3.5** | 자신의 기획 편향을 인식하기 시작한다. "나는 항상 기능 중심으로 생각하는 경향이 있어". | 자기 인식 패턴 |
| **L4.0** | 기획의 트레이드오프를 분석적으로 논의한다. "확장성 vs 집중도의 균형을 잡아야 해". | strategic_ratio > 0.05, metacognitive_ratio > 0 |
| **L4.5** | 자신의 기획 패턴을 의식적으로 개선하려 한다. "이전 기획에서는 리스크 분석이 부족했으니 이번엔 강화할게". | 자기 참조 패턴 |
| **L5.0** | 시스템적 사고를 적용한다. 2차, 3차 효과를 예측하고 전략에 반영한다. "이 전략이 성공하면 경쟁자는 어떻게 반응할까?" | strategic_ratio > 0.1, 시스템 사고 키워드 |
| **L5.5** | 자신의 기획 데이터를 분석한다. 어떤 프레임워크가 어떤 상황에서 효과적이었는지 정량적으로 파악한다. | 데이터 분석 패턴 |
| **L6.0** | 기획 방법론을 체계화하고 문서화한다. 조직에 전략적 사고 프레임워크를 교육한다. | 방법론 키워드, 교육/공유 패턴 |
| **L6.5** | 다양한 기획 전략을 비교 실험하고 최적의 접근법을 도출한다. | A/B 비교, 실험 키워드 |
| **L7.0** | AI 활용 기획의 새로운 패러다임이나 의사결정 프레임워크를 창안하고 산업에 기여한다. | 외부 기여 증거 |

## 레벨 판정 공식

### Fit Score 계산

각 세션에서 레벨 L에 대한 적합도 점수(Fit Score)를 계산한다:

```
F_L = DECOMP_score × 0.20
    + VERIFY_score × 0.15
    + ORCH_score   × 0.15
    + FAIL_score   × 0.15
    + CTX_score    × 0.15
    + META_score   × 0.20

각 차원 score = 해당 레벨 행동 기술자의 자동 감지 신호 충족 비율 (0.0~1.0)
```

### 최종 레벨 결정

```
1. 각 차원에서 정수 레벨과 0.5 레벨 기준 충족도 계산
2. 가중 평균으로 raw_score 산출 (소수점 2자리)
3. 0.5 단위로 반올림하여 공식 레벨 결정

반올림 기준:
  x.00 ~ x.24 → x.0
  x.25 ~ x.74 → x.5
  x.75 ~ x.99 → (x+1).0
```

### 게이트 조건

Fit Score가 아무리 높아도 게이트를 통과하지 못하면 해당 레벨로 판정되지 않는다.

| 게이트 | 필수 조건 | 근거 |
|--------|----------|------|
| L3 진입 | context_specificity_ratio > 0.5 | 페르소나/타깃 없이 L3 불가 |
| L4 진입 | verification_ratio > 0.15 AND correction_ratio > 0.05 | 기획 검증 없이 L4 불가 |
| L5 진입 | (tool_diversity > 8 OR has_orchestration) AND strategic_ratio > 0.05 | 시스템적 사고 없이 L5 불가 |
| L6 진입 | teamName 존재 OR orch_tool_count > 10 | 멀티에이전트 기획 경험 필수 |
| L7 진입 | L6 게이트 + 외부 기여 증거 | 산업/커뮤니티 기여 필수 |

### 3회 일관성 원칙

일회성 성과는 인정하지 않는다. NTRP가 "일관되게 할 수 있는 것"을 기준으로 레벨을 판정하듯:

```
승급 조건:
  최근 5개 세션 중 3개 이상에서 상위 레벨 Fit Score가 현 레벨보다 높아야 함
```

## 승급/강등 기준

### NTRP 비대칭 원칙

심리적 안전과 동기 부여를 위해, 승급은 빠르게, 강등은 느리게 설계한다.

| 방향 | 조건 | 사용자 경험 |
|------|------|-----------|
| **승급** | 2회 연속 상위 레벨 Fit Score → **즉시 승급** | 즉각적 보상, 성취감 |
| **강등** | 3회 연속 하위 레벨 Fit Score + **사용자 동의** | 동의 없이 강등하지 않음 |

### 80% 경계선 (L4→L5) 특별 취급

L4→L5는 "기획서를 만드는 사람"(Planner)에서 "전략을 설계하는 사람"(Strategist)으로 전환하는 핵심 순간이다.

- **승급 조건 강화**: L4→L5는 3회 연속 (일반 승급은 2회)
- **승급 시 특별 이벤트**: "벽 돌파" 메시지와 함께 시스템적 사고 성장 증거를 상세히 표시
- **의미**: "기획하는 사람"에서 "전략가"로의 전환

### Strike 시스템 (과소평가 방지)

자기평가보다 실측이 높으면 자동 상향한다:

```
if 동적_레이팅 > 자기평가_레벨 + 1.0 (3회 연속):
    Strike 발생 → 3 Strikes면 자동 승급
```

### 바닥 효과 보정

- 첫 세션 완료 즉시 최소 L1.5 부여
- 3세션 이내에 기본 도구 2종 이상 사용 시 L2.0으로 자동 승급
- L1은 "AI를 처음 접하는 순간"에만 해당

### 정체기 대응

같은 레벨에 4주 이상 머무르면 마이크로 마일스톤을 제공한다:

```
예시 (L3 → L4 정체 시):
  [ ] 대안 비교가 포함된 기획 3건 완성
  [ ] 가설 명시 + 검증 요청 포함 3세션
  [ ] 이해관계자 관점 포함 기획 2건

  진행도: ██████████░░░░░░ 2/3 완료
```
