# Operator 성장 추적 & 레벨 시스템 v2.0

> 업무 프로세스를 자동화하고 효율화하는 사람의 AI 활용 성장을 측정하는 체계

---

## 레벨 아이콘과 별명

| 레벨 | 영문명 | 한국어 별명 | 아이콘 | 서사 단계 |
|------|--------|------------|--------|----------|
| L1 | User | 사용자 | ○ | **수동** |
| L2 | Repeater | 반복자 | ◐ | 수동→능동 |
| L3 | Optimizer | 최적화자 | ● | **능동** |
| L4 | Builder | 구축자 | ◆ | 능동→주도 |
| L5 | Engineer | 엔지니어 | ⬡ | **주도** |
| L6 | Orchestrator | 오케스트레이터 | ✦ | 주도→창조 |
| L7 | Automator | 자동화 마스터 | ◈ | **창조** |

**불꽃 성장 메타포**: 씨앗불(○) → 반쪽불(◐) → 온전한 불(●) → 보석불(◆) → 벌집불(⬡) → 별불(✦) → 다이아불(◈)

```
제1막: 관찰과 모방 (L1~L2)     제2막: 참여와 성장 (L3~L5)     제3막: 주도와 혁신 (L6~L7)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
AI가 이끈다                     사용자와 AI가 협력한다            사용자가 AI를 이끈다
도구를 사용한다                 워크플로우를 설계한다             시스템을 운영한다
```

## 6대 기술 차원 정의

바선생은 AI 활용 기술을 6개 독립 축으로 측정한다. 테니스 NTRP가 포핸드/백핸드/서브/발리/풋워크/전략을 각각 평가하듯, 각 축을 따로 보고 종합한다.

| 코드 | 기술 차원 | Operator에서의 의미 |
|------|----------|---------------------|
| **DECOMP** | 작업 분해 | 복잡한 업무 프로세스를 자동화 가능한 단위로 쪼개는 능력. 수동 작업을 단계별로 분해하여 자동화 대상을 식별하는 것. |
| **VERIFY** | 검증 전략 | 자동화 결과를 확인하고, 에러 처리를 설계하고, 품질을 보장하는 능력. Operator의 핵심 축. |
| **ORCH** | 오케스트레이션 | API, 스크립트, 노코드 도구, 에이전트를 조합하여 업무 파이프라인을 구축하는 능력. Operator의 핵심 축. |
| **FAIL** | 실패 대응 | 자동화 장애, 예외 상황, 데이터 오류에 체계적으로 대응하는 능력. Operator의 핵심 축. |
| **CTX** | 맥락 관리 | 업무 프로세스의 입력/출력, 의존관계, 제약 조건을 AI에게 제공하는 능력. |
| **META** | 메타인지 | 자신의 자동화 전략을 인식하고 ROI를 판단하는 능력. |

## Operator 가중치

Operator는 **운영하는 사람**이다. 도구를 조합하고(ORCH), 오류에 대응하고(FAIL), 결과를 검증하는(VERIFY) 것이 성장의 70%를 차지한다. 안정적으로 돌아가는 시스템을 만드는 것이 목표다.

| 기술 차원 | 가중치 | 이유 |
|----------|--------|------|
| **DECOMP** | 20% | 프로세스를 자동화 단위로 분해하는 것이 시작점 |
| **VERIFY** | **25%** | 자동화가 제대로 돌아가는지 검증하는 것이 운영의 핵심 |
| **ORCH** | **25%** | 여러 도구와 API를 조합하는 것이 Operator의 1번 기술 |
| **FAIL** | 15% | 장애 대응은 필수이나 ORCH/VERIFY에 포함되는 부분도 있음 |
| **CTX** | 10% | 프로세스 맥락 제공은 기본기 |
| **META** | 5% | 전략적 사고는 L5 이상에서 본격적으로 필요 |

```
DECOMP ██████████████████████████      20%
VERIFY ████████████████████████████████ 25%
ORCH   ████████████████████████████████ 25%
FAIL   ████████████████████            15%
CTX    █████████████                   10%
META   ███████                          5%
```

## 7단계 레벨 정의

### Level 1 (L1.0) — User (사용자) ○

**NTRP 대응**: 1.5 (공을 코트 안에 넣는 것이 목표)
**핵심 특성**: 다른 사람이 만들어둔 자동화 도구를 사용한다. 반복 작업을 인식하지 못하거나, 자동화라는 개념 자체가 낯설다. "컴퓨터가 알아서 해주면 좋겠다"고 막연히 생각한다.
**판별 질문**: "매주 반복하는 작업을 3가지 말할 수 있나요?" → "없다" / "잘 모르겠다"
**인지 수준**: Remember (기억) — 나열하다, 인식하다, 따라하다

### Level 1.5 (L1.5) — User+ ○

**전환 중 행동**: 반복 작업의 존재를 인식하기 시작한다. "이거 매번 하는 건데..."라는 생각이 들지만 자동화 시도는 아직 없다.

### Level 2 (L2.0) — Repeater (반복자) ◐

**NTRP 대응**: 2.5 (느린 랠리 지속 가능)
**핵심 특성**: 반복 작업을 인식하고 자동화를 시도한다. Zapier, Make 같은 노코드 도구로 간단한 연결을 만든다. 에러 처리 없이 "일단 돌아가면 된다" 수준.
**판별 질문**: "반복 작업을 자동화한 적 있나요?" → "간단한 것 하나 만들어봤다"
**인지 수준**: Understand (이해) — 설명하다, 비교하다, 질문하다

### Level 2.5 (L2.5) — Repeater+ ◐

**전환 중 행동**: 자동화가 실패했을 때 "왜 안 되지?"라고 궁금해하기 시작한다. AI에게 자동화 도움을 요청하기 시작한다.

### Level 3 (L3.0) — Optimizer (최적화자) ●

**NTRP 대응**: 3.0 (중간 속도 랠리 안정적)
**핵심 특성**: 여러 단계로 이루어진 워크플로우를 설계하고 연결한다. AI와 함께 간단한 스크립트를 작성한다. 기본적인 에러 처리를 시도하지만 체계적이지는 않다. 여러 API를 연결한다.
**판별 질문**: "다단계 워크플로우를 설계한 적 있나요?" → "있다, API도 연결해봤다"
**인지 수준**: Apply (적용) — 적용하다, 실행하다, 구현하다

### Level 3.5 (L3.5) — Optimizer+ ●

**전환 중 행동**: 에러 로그의 필요성을 인식한다. 환경 변수로 비밀 정보를 분리하기 시작하지만 아직 일관적이지 않다.

### Level 4 (L4.0) — Builder (구축자) ◆

**NTRP 대응**: 4.0 (포핸드, 백핸드 모두 방향+깊이 제어)
**핵심 특성**: 에러 처리, 로깅, 재사용성을 고려한 자동화를 설계한다. 보안을 인식하고 환경 변수를 사용한다. 자동화 결과를 모니터링하고 개선한다. 재사용 가능한 모듈식 자동화를 만든다.
**판별 질문**: "자동화에 에러 처리와 로깅이 있나요?" → "대부분 있다"
**인지 수준**: Analyze (분석) — 분석하다, 구별하다, 비판하다

### Level 4.5 (L4.5) — Builder+ ◆

**전환 중 행동**: 완전 무인화를 목표로 하기 시작한다. 모니터링 대시보드를 구성하려는 시도가 보인다. 자동화 문서화를 습관화하기 시작한다.

### Level 5 (L5.0) — Engineer (엔지니어) ⬡

**NTRP 대응**: 5.0 (예측 능력 발달, 고급 샷 구사)
**핵심 특성**: 완전 무인화 시스템을 구축하고 운영한다. 이벤트 드리븐 아키텍처를 설계한다. 자동화의 ROI를 계산하고 우선순위를 결정한다. 다른 사람에게 자동화 가이드를 제공한다.
**판별 질문**: "사람 개입 없이 한 달 이상 돌아가는 자동화가 있나요?" → "3개 이상 있다"
**인지 수준**: Evaluate (평가) — 평가하다, 비교하다, 판단하다

> **L4→L5는 "80%의 벽" 돌파 지점이다.** "돌아가게 만드는 사람"(Builder)에서 "안정적으로 운영하는 사람"(Engineer)으로 전환하는 순간.

### Level 5.5 (L5.5) — Engineer+ ⬡

**전환 중 행동**: 조직 전체의 자동화 현황을 파악하려 한다. 자동화 표준이나 가이드라인을 작성하기 시작한다. ROI 기반 우선순위 결정을 체계화하려는 시도가 보인다.

### Level 6 (L6.0) — Orchestrator (오케스트레이터) ✦

**NTRP 대응**: 6.0 (세미프로, 전국 수준)
**핵심 특성**: 조직 전체의 자동화 전략을 수립하고 주도한다. 멀티에이전트 시스템을 설계하고 운영한다. 자동화 표준과 가이드라인을 수립한다. 복잡한 이벤트 드리븐 시스템을 설계한다.
**판별 질문**: "조직 수준의 자동화 전략을 수립한 적 있나요?" → "수립하고 운영 중이다"
**인지 수준**: Create (창조-응용) — 설계하다, 구성하다, 통합하다

### Level 6.5 (L6.5) — Orchestrator+ ✦

**전환 중 행동**: 자신만의 자동화 방법론을 정립한다. 팀에 자동화 교육을 실시한다. 커스텀 도구와 프레임워크를 개발한다.

### Level 7 (L7.0) — Automator (자동화 마스터) ◈

**NTRP 대응**: 7.0 (세계 최상위)
**핵심 특성**: AI 활용 자동화의 새로운 방법론이나 패러다임을 창조한다. 커뮤니티/산업에 기여한다. 자동화 도구 생태계에 기여한다.
**판별 질문**: "다른 사람들이 참고하는 자동화 방법론, 도구, 프레임워크를 만든 적 있나요?" → "만들었고, 다른 사람들이 사용하고 있다"
**인지 수준**: Create (창조-혁신) — 창조하다, 발명하다, 혁신하다

## 6축 x 7레벨 행동 기술자 매트릭스

각 칸에는 **관찰 가능한 행동**(~를 한다)과 **자동 감지 신호**를 함께 기술한다. 0.5 레벨은 전환 중 행동이다.

### DECOMP (작업 분해) — Operator의 프로세스 분석

> 복잡한 업무 프로세스를 자동화 가능한 단위로 쪼개는 행동

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | 업무를 통째로 요청한다. "이 보고서 만들어줘" 수준. 프로세스 분해 개념 없음. | avg_msg_len < 100자, 번호 목록 비율 = 0% |
| **L1.5** | 가끔 "먼저 데이터를 가져와서"라는 표현이 등장하지만 체계적이지 않다. | avg_msg_len 100~200자, 순서 표현 비율 < 5% |
| **L2.0** | 업무를 여러 단계로 나열한다. "데이터 가져오고, 정리하고, 보고서 만들어줘". 자동화 순서는 미고려. | avg_msg_len 150~300자, 나열형 요청 비율 > 10% |
| **L2.5** | "먼저 API로 데이터를 가져오고, 그 다음 가공해줘" 형태의 순서 지정이 나타난다. | 순서 접속사 포함 비율 > 15% |
| **L3.0** | 워크플로우를 3단계 이상으로 분해한다. 트리거 → 처리 → 출력의 기본 구조. | 번호 목록 포함 비율 > 20%, 단계 수 평균 3+ |
| **L3.5** | 자동화 단계 간 데이터 흐름을 인식한다. "1단계 출력이 2단계 입력이 된다". | 데이터 흐름 표현, 조건 표현 > 10% |
| **L4.0** | 업무를 모듈 단위로 분해한다. 각 모듈의 입력/출력/에러 조건을 명시한다. 재사용 가능한 단위로 설계한다. | 모듈/함수 단위 지시 > 40%, 분해 단계 4+ |
| **L4.5** | 자동화 가능한 부분과 수동 개입이 필요한 부분을 명확히 구분한다. | 자동화/수동 구분 표현 > 20% |
| **L5.0** | 전체 업무 프로세스를 이벤트 드리븐 아키텍처로 설계한다. 트리거, 핸들러, 콜백을 구분한다. | 프로세스 구조 언급 > 30%, 다중 세션 연속 |
| **L5.5** | 각 자동화 단계를 다른 에이전트/서비스에 배분한다. 마이크로서비스 개념 적용. | has_orchestration = True, orch_tool_count > 3 |
| **L6.0** | 조직 전체의 업무 자동화 아키텍처를 설계한다. 시스템 간 연동, 데이터 파이프라인을 구축한다. | teamName 존재, orch_tool_count > 10 |
| **L6.5** | 자동 분해 템플릿, 프로세스 분석 프레임워크를 만들어 조직에 배포한다. | 커스텀 명령어/스킬 사용 |
| **L7.0** | 새로운 업무 프로세스 분해 방법론이나 자동화 아키텍처 패턴을 창안한다. | 외부 공유 증거 |

### VERIFY (검증 전략) — Operator의 품질 보증

> 자동화 결과를 확인하고, 에러 처리를 설계하고, 품질을 보장하는 행동. Operator의 핵심 축.

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | 자동화 결과를 확인하지 않는다. "됐겠지"하고 넘어간다. | verification_ratio = 0 |
| **L1.5** | 결과가 이상할 때만 확인한다. "어? 이거 왜 이래?" | correction_ratio < 0.05 |
| **L2.0** | 자동화 실행 후 결과를 수동으로 확인한다. "돌렸는데 맞게 나왔나 봐줘". | verification_ratio > 0, followup_ratio > 0.03 |
| **L2.5** | 특정 결과값을 지적하며 확인한다. "이 숫자가 맞아? 지난번이랑 다른데". | correction_ratio > 0.05 |
| **L3.0** | 자동화에 기본 검증 단계를 포함한다. "실행 후 결과 건수를 확인해". | verification_ratio > 0.15, 검증 단계 언급 |
| **L3.5** | 에러 처리를 요청하기 시작한다. "실패하면 알림 보내줘", "에러 로그 남겨줘". | "에러"/"로그"/"알림" 키워드 > 5% |
| **L4.0** | 체계적 에러 처리를 설계한다. try-catch, 재시도, 롤백을 포함한다. 실행 결과를 로깅한다. | verification_ratio > 0.2, correction_ratio > 0.1 |
| **L4.5** | 자동화의 엣지 케이스를 사전에 정의한다. "데이터가 비어있으면?", "API가 타임아웃되면?" | alternative_ratio > 0.05, 엣지 케이스 키워드 |
| **L5.0** | 모니터링 대시보드를 설계한다. 자동화 건전성 지표(실행 횟수, 성공률, 지연 시간)를 추적한다. | constraint_ratio > 0.1, 모니터링 키워드 |
| **L5.5** | 자동 알림 체계를 구축한다. 임계값 초과 시 경고, 장애 시 에스컬레이션을 설계한다. | 자동화 도구 사용, 알림 체계 키워드 |
| **L6.0** | 조직 수준의 자동화 품질 관리 프로세스를 설계한다. SLA, 품질 게이트, 감사 로그를 구축한다. | 시스템 수준 검증 키워드 |
| **L6.5** | 자동화 품질 메트릭을 추적하고 트렌드를 분석하여 시스템을 개선한다. | 메트릭 추적/분석 패턴 |
| **L7.0** | 새로운 자동화 품질 보증 방법론이나 운영 프레임워크를 창안한다. | 외부 공유 증거 |

### ORCH (오케스트레이션) — Operator의 시스템 통합

> API, 스크립트, 노코드 도구, 에이전트를 조합하여 업무 파이프라인을 구축하는 행동. Operator의 핵심 축.

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | 단일 SaaS 도구의 기본 기능만 사용한다. 노션, 스프레드시트 등. | tool_diversity <= 2 |
| **L1.5** | 도구의 부가 기능(자동 정렬, 필터 등)을 가끔 사용한다. | tool_diversity = 2~3 |
| **L2.0** | 노코드 도구(Zapier, Make)로 단순 트리거-액션을 연결한다. "이메일 오면 스프레드시트에 기록". | tool_diversity = 3~4, 노코드 도구 사용 |
| **L2.5** | 2~3개 서비스를 연결한다. 수동 복사-붙여넣기를 자동화로 대체하기 시작한다. | tool_diversity = 4 |
| **L3.0** | 4~6개 도구와 API를 연결하여 다단계 워크플로우를 구성한다. AI와 함께 스크립트를 작성한다. | tool_diversity = 4~6, tools_per_turn > 3 |
| **L3.5** | 도구 선택 이유를 인식한다. "이건 Zapier보다 Python 스크립트가 더 유연해". | 도구 선택 이유 언급, tool_diversity > 5 |
| **L4.0** | 7~9개 도구를 통합한다. API 키 관리, 환경 변수 분리, 모듈식 설계. | tool_diversity = 7~9, tools_per_turn > 5 |
| **L4.5** | 반복 가능한 자동화 패턴을 보유한다. "데이터 수집 → 가공 → 저장 → 알림"이 표준화됨. | 반복 도구 패턴, tool_diversity > 8 |
| **L5.0** | 이벤트 드리븐 시스템을 구축한다. 웹훅, 크론잡, 큐를 활용한 비동기 처리. | has_orchestration = True, orch_tool_count > 5 |
| **L5.5** | 마이크로서비스 구조로 자동화를 설계한다. 각 서비스가 독립적으로 배포/확장 가능. | orch_tool_count > 8, 마이크로서비스 키워드 |
| **L6.0** | 조직 전체의 자동화 인프라를 설계한다. 서비스 메시, API 게이트웨이, 중앙 모니터링. | teamName 존재, orch_tool_count > 15 |
| **L6.5** | 커스텀 MCP 서버, 내부 도구, CI/CD를 통합한 자동화 플랫폼을 구축한다. | mcp_tool_count > 5, bypass_ratio > 0.7 |
| **L7.0** | 새로운 자동화 오케스트레이션 패턴이나 플랫폼을 창안한다. 도구 생태계에 기여한다. | tool_diversity > 15, 외부 기여 증거 |

### FAIL (실패 대응) — Operator의 장애 대응

> 자동화 장애, 예외 상황, 데이터 오류에 체계적으로 대응하는 행동

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | 자동화가 실패하면 인식하지 못한다. 결과를 확인하지 않아서 실패 자체를 모른다. | 에러 인식 패턴 없음 |
| **L1.5** | 실패를 인식하지만 "다시 돌려봐"만 요청한다. 원인 분석 없음. | 동일 요청 반복 비율 > 30% |
| **L2.0** | 에러 메시지를 읽고 "이게 뭐야?"라고 물어본다. AI에게 해석을 맡긴다. | "에러"/"error" + 해석 요청 패턴 |
| **L2.5** | 에러 위치를 파악한다. "API 호출 단계에서 실패했어". | 에러 위치 지적 패턴 |
| **L3.0** | 에러 원인을 추측하고 가설을 제시한다. "API 키가 만료된 것 같아". | 원인 추측 패턴 > 10% |
| **L3.5** | 에러 해결 후 재발 방지를 고려한다. "이 에러가 다시 안 나게 하려면?" | 재발 방지 키워드 > 5% |
| **L4.0** | 에러 유형별 대응 전략을 설계한다. 재시도, 대체 경로, 수동 개입 기준을 정의한다. | alternative_ratio > 0.05, 에러 대응 패턴 |
| **L4.5** | 장애 예방 설계를 한다. 입력 검증, 타임아웃 설정, 서킷 브레이커를 구현한다. | 예방 키워드("미리", "방지", "timeout") |
| **L5.0** | 장애 대응을 자동화한다. 자동 재시도, 자동 롤백, 자동 알림. 장애 등급을 분류한다. | 자동 복구 패턴, recovery 성공률 > 70% |
| **L5.5** | 카오스 엔지니어링 개념을 적용한다. 의도적으로 장애를 주입하여 탄력성을 테스트한다. | 테스트 키워드, 탄력성 패턴 |
| **L6.0** | 조직 수준의 장애 대응 프로세스를 설계한다. 인시던트 매뉴얼, 에스컬레이션 체계, SLA. | 시스템 수준 장애 대응 키워드 |
| **L6.5** | 장애 데이터를 수집하고 분석한다. MTTR(평균 복구 시간) 추적, 사후 분석(post-mortem) 문화. | 분석/메트릭 추적 패턴 |
| **L7.0** | 새로운 자동화 탄력성 패턴이나 장애 대응 방법론을 창안한다. | 외부 기여 증거 |

### CTX (맥락 관리) — Operator의 프로세스 명세

> 업무 프로세스의 입력/출력, 의존관계, 제약 조건을 AI에게 제공하는 행동

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | 맥락 없이 요청한다. "자동화해줘". 어떤 업무인지, 데이터가 어디 있는지 설명 없음. | context_specificity_ratio < 0.3 |
| **L1.5** | 가끔 목적을 언급한다. "매주 하는 보고서 작업을 자동화하고 싶은데". | 목적 표현 비율 < 10% |
| **L2.0** | 업무 흐름을 간략히 설명한다. "이메일에서 데이터를 가져와서 스프레드시트에 넣는 작업". | context_specificity_ratio > 0.4 |
| **L2.5** | 도구명이나 서비스명을 구체적으로 언급한다. "Gmail API로", "Google Sheets에". | context_specificity_ratio > 0.5 |
| **L3.0** | 입력/출력 데이터 형식을 명시한다. "JSON 형태로 받아서 CSV로 변환". API 엔드포인트를 제공한다. | context_specificity_ratio > 0.6, 데이터 형식 키워드 |
| **L3.5** | 제약 조건을 명시한다. "매일 오전 9시에 실행", "100건 이상일 때만", "비밀번호는 환경변수로". | constraint_ratio > 0.03 |
| **L4.0** | 비기능 요구사항을 포함한다. 실행 시간 제한, 동시성 제약, 데이터 보안 요건. | constraint_ratio > 0.05, 비기능 키워드 |
| **L4.5** | 시스템 아키텍처 맥락을 제공한다. "이 서비스는 X와 연동되고, Y 데이터베이스를 사용해". | 아키텍처 키워드, avg_msg_len > 600 |
| **L5.0** | 운영 환경 전체를 맥락으로 제공한다. 인프라, 의존성, 배포 전략을 포함한다. | 인프라 키워드, 맥락 파일 참조 |
| **L5.5** | 맥락 제공을 자동화한다. 인프라 문서, API 명세, 환경 설정을 사전 준비한다. | 설정 파일/템플릿 사용 |
| **L6.0** | 조직 수준의 인프라 문서화 체계를 구축한다. 서비스 카탈로그, API 레지스트리를 설계한다. | 구조화된 맥락 파일 다수 |
| **L6.5** | 맥락 품질을 최적화한다. 어떤 인프라 정보가 자동화 품질에 가장 큰 영향을 미치는지 분석한다. | 맥락 최적화/A/B 비교 |
| **L7.0** | 새로운 자동화 맥락 관리 방법론(인프라 as 코드, 서비스 메시 등)을 창안한다. | 외부 기여 증거 |

### META (메타인지) — Operator의 자동화 전략

> 자신의 자동화 전략을 인식하고 ROI를 판단하는 행동

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | 자동화 방식에 대한 성찰 없음. "그냥 해본다". | metacognitive_ratio = 0 |
| **L1.5** | "자동화 편하네" / "어렵네"라는 수준의 인상적 평가. | 평가 표현 있으나 기준 없음 |
| **L2.0** | 어떤 작업이 자동화에 적합한지 경험적으로 느낀다. | 경험적 평가 패턴 |
| **L2.5** | 자동화 방식(노코드 vs 스크립트)에 따라 결과가 다르다는 것을 인식한다. | 도구 비교 패턴 |
| **L3.0** | 다른 사람의 자동화 방법을 참고한다. 템플릿이나 레시피를 활용한다. | 참고/템플릿 키워드 |
| **L3.5** | 어떤 도구가 어떤 상황에 적합한지 경험적으로 구분한다. | 상황별 도구 선택 패턴 |
| **L4.0** | 자동화의 비용 대비 효과를 분석한다. "이 자동화로 주당 X시간을 절약할 수 있다". | strategic_ratio > 0.05, ROI 키워드 |
| **L4.5** | 자동화 우선순위를 의식적으로 결정한다. "높은 ROI 순서대로 자동화하겠다". | 우선순위 키워드, 자기 참조 패턴 |
| **L5.0** | 자동화 전략을 사전에 수립한다. "올해 자동화 로드맵은..." | strategic_ratio > 0.1, 전략/로드맵 키워드 |
| **L5.5** | 자동화 성과를 데이터로 추적한다. 절약 시간, 에러율, 안정성 트렌드를 분석한다. | 데이터 분석 패턴 |
| **L6.0** | 조직 자동화 전략을 체계화하고 문서화한다. 팀에 교육한다. | 방법론/교육 키워드 |
| **L6.5** | 다양한 자동화 전략을 비교 실험하고 최적의 접근법을 도출한다. | A/B 비교, 실험 키워드 |
| **L7.0** | AI 활용 자동화의 새로운 패러다임이나 운영 프레임워크를 창안하고 산업에 기여한다. | 외부 기여 증거 |

## 레벨 판정 공식

### Fit Score 계산

각 세션에서 레벨 L에 대한 적합도 점수(Fit Score)를 계산한다:

```
F_L = DECOMP_score × 0.20
    + VERIFY_score × 0.25
    + ORCH_score   × 0.25
    + FAIL_score   × 0.15
    + CTX_score    × 0.10
    + META_score   × 0.05

각 차원 score = 해당 레벨 행동 기술자의 자동 감지 신호 충족 비율 (0.0~1.0)
```

### 최종 레벨 결정

```
1. 각 차원에서 정수 레벨과 0.5 레벨 기준 충족도 계산
2. 가중 평균으로 raw_score 산출 (소수점 2자리)
3. 0.5 단위로 반올림하여 공식 레벨 결정

반올림 기준:
  x.00 ~ x.24 → x.0
  x.25 ~ x.74 → x.5
  x.75 ~ x.99 → (x+1).0
```

### 게이트 조건

Fit Score가 아무리 높아도 게이트를 통과하지 못하면 해당 레벨로 판정되지 않는다.

| 게이트 | 필수 조건 | 근거 |
|--------|----------|------|
| L3 진입 | context_specificity_ratio > 0.5 | 입출력 명세 없이 L3 불가 |
| L4 진입 | verification_ratio > 0.15 AND correction_ratio > 0.05 | 에러 처리 없이 L4 불가 |
| L5 진입 | (tool_diversity > 8 OR has_orchestration) AND strategic_ratio > 0.05 | 시스템 통합과 전략적 사고 필수 |
| L6 진입 | teamName 존재 OR orch_tool_count > 10 | 멀티에이전트 운용 경험 필수 |
| L7 진입 | L6 게이트 + 외부 기여 증거 | 산업/커뮤니티 기여 필수 |

### 3회 일관성 원칙

일회성 성과는 인정하지 않는다. NTRP가 "일관되게 할 수 있는 것"을 기준으로 레벨을 판정하듯:

```
승급 조건:
  최근 5개 세션 중 3개 이상에서 상위 레벨 Fit Score가 현 레벨보다 높아야 함
```

## 승급/강등 기준

### NTRP 비대칭 원칙

심리적 안전과 동기 부여를 위해, 승급은 빠르게, 강등은 느리게 설계한다.

| 방향 | 조건 | 사용자 경험 |
|------|------|-----------|
| **승급** | 2회 연속 상위 레벨 Fit Score → **즉시 승급** | 즉각적 보상, 성취감 |
| **강등** | 3회 연속 하위 레벨 Fit Score + **사용자 동의** | 동의 없이 강등하지 않음 |

### 80% 경계선 (L4→L5) 특별 취급

L4→L5는 "돌아가게 만드는 사람"(Builder)에서 "안정적으로 운영하는 사람"(Engineer)으로 전환하는 핵심 순간이다.

- **승급 조건 강화**: L4→L5는 3회 연속 (일반 승급은 2회)
- **승급 시 특별 이벤트**: "벽 돌파" 메시지와 함께 자동화 안정성 증거를 상세히 표시
- **의미**: "만드는 자동화"에서 "운영하는 시스템"으로의 전환

### Strike 시스템 (과소평가 방지)

자기평가보다 실측이 높으면 자동 상향한다:

```
if 동적_레이팅 > 자기평가_레벨 + 1.0 (3회 연속):
    Strike 발생 → 3 Strikes면 자동 승급
```

### 바닥 효과 보정

- 첫 세션 완료 즉시 최소 L1.5 부여
- 3세션 이내에 기본 도구 2종 이상 사용 시 L2.0으로 자동 승급
- L1은 "AI를 처음 접하는 순간"에만 해당

### 정체기 대응

같은 레벨에 4주 이상 머무르면 마이크로 마일스톤을 제공한다:

```

  [ ] 모든 자동화에 로그 설정
  [ ] 환경 변수 분리 100% 달성

  진행도: ██████████░░░░░░ 2/3 완료
```
