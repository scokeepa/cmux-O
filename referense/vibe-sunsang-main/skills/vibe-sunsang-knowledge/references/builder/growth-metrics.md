# Builder 성장 추적 & 레벨 시스템 v2.0

> 코드를 작성하여 제품을 만드는 사람의 AI 활용 성장을 측정하는 체계

## 레벨 아이콘과 별명

| 레벨 | 영문명 | 한국어 별명 | 아이콘 | 서사 단계 |
|------|--------|------------|--------|----------|
| L1 | Observer | 관찰자 | ○ | **수동** |
| L2 | Tinkerer | 만지작거리는 사람 | ◐ | 수동→능동 |
| L3 | Collaborator | 협력자 | ● | **능동** |
| L4 | Pilot | 조종사 | ◆ | 능동→주도 |
| L5 | Architect | 설계자 | ⬡ | **주도** |
| L6 | Conductor | 지휘자 | ✦ | 주도→창조 |
| L7 | Forgemaster | 대장장이 | ◈ | **창조** |

**불꽃 성장 메타포**: 씨앗불(○) → 반쪽불(◐) → 온전한 불(●) → 보석불(◆) → 벌집불(⬡) → 별불(✦) → 다이아불(◈)

```
제1막: 관찰과 모방 (L1~L2)     제2막: 참여와 성장 (L3~L5)     제3막: 주도와 혁신 (L6~L7)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
AI가 이끈다                     사용자와 AI가 협력한다            사용자가 AI를 이끈다
결과를 받아들인다               과정에 개입한다                   시스템을 설계한다
```

## 6대 기술 차원 정의

바선생은 AI 활용 기술을 6개 독립 축으로 측정한다. 테니스 NTRP가 포핸드/백핸드/서브/발리/풋워크/전략을 각각 평가하듯, 각 축을 따로 보고 종합한다.

| 코드 | 기술 차원 | Builder에서의 의미 |
|------|----------|-------------------|
| **DECOMP** | 작업 분해 | "이 앱 만들어줘"를 모듈/파일/함수 단위로 쪼개서 지시하는 능력. Builder의 핵심 중 핵심. |
| **VERIFY** | 검증 전략 | AI가 생성한 코드를 테스트하고, 엣지 케이스를 확인하고, 코드 리뷰를 요청하는 습관. Builder의 또 다른 핵심. |
| **ORCH** | 오케스트레이션 | 여러 도구(Bash, Read, Edit, Grep 등)와 에이전트를 조합하여 개발 워크플로우를 구성하는 능력. |
| **FAIL** | 실패 대응 | 빌드 에러, 런타임 오류, 예상 밖의 결과에 원인을 분석하고 체계적으로 대응하는 능력. |
| **CTX** | 맥락 관리 | 파일 경로, 기존 코드 구조, 기술 스택, 제약 조건을 AI에게 제공하는 능력. |
| **META** | 메타인지 | "나는 AI를 어떻게 쓰고 있는가?"를 인식하고, 접근 방식을 전략적으로 조정하는 능력. |

## Builder 가중치

Builder는 **만드는 사람**이다. 작업을 잘 쪼개고(DECOMP), 결과를 잘 검증하는(VERIFY) 것이 성장의 절반이다.

| 기술 차원 | 가중치 | 이유 |
|----------|--------|------|
| **DECOMP** | **25%** | 복잡한 기능을 모듈/파일/함수로 분해하는 것이 Builder의 1번 기술 |
| **VERIFY** | **25%** | 코드가 실제로 동작하는지 확인하는 것이 Builder의 2번 기술 |
| **ORCH** | 15% | 개발 도구를 조합하는 능력도 중요하지만 핵심은 아님 |
| **FAIL** | 15% | 에러 대응은 필수이나 분해/검증에 비해 보조적 |
| **CTX** | 10% | 맥락 제공은 기본기에 해당 |
| **META** | 10% | 전략적 사고는 L5 이상에서 본격적으로 필요 |

```
DECOMP ████████████████████████████████ 25%
VERIFY ████████████████████████████████ 25%
ORCH   ████████████████████            15%
FAIL   ████████████████████            15%
CTX    █████████████                   10%
META   █████████████                   10%
```

## 7단계 레벨 정의

### Level 1 (L1.0) — Observer (관찰자) ○

**NTRP 대응**: 1.5 (공을 코트 안에 넣는 것이 목표)
**핵심 특성**: AI 도구가 무엇인지 알고, 안내에 따라 처음 사용해본다. 결과를 그대로 수용한다.
**판별 질문**: "AI에게 요청할 때, 원하는 결과를 얻지 못하면 어떻게 하나요?" → "다시 물어본다" / "포기한다"
**인지 수준**: Remember (기억) — 나열하다, 인식하다, 따라하다

### Level 1.5 (L1.5) — Observer+ ○

**전환 중 행동**: L1에서 L2로 넘어가는 과도기. 가끔 "왜?"를 물어보기 시작하지만 아직 습관화되지 않음. 결과를 수용하면서도 불만족을 표현할 때가 있다.

### Level 2 (L2.0) — Tinkerer (만지작거리는 사람) ◐

**NTRP 대응**: 2.5 (느린 랠리 지속 가능)
**핵심 특성**: 기본 기능을 독립적으로 사용한다. 결과에 대해 "왜?"를 물어보기 시작한다. 에러 메시지를 읽어보려 시도한다.
**판별 질문**: "마지막으로 AI가 준 답변에 추가 질문을 한 적이 있나요?" → "가끔 한다"
**인지 수준**: Understand (이해) — 설명하다, 비교하다, 질문하다

### Level 2.5 (L2.5) — Tinkerer+ ◐

**전환 중 행동**: 요청에 배경 설명을 추가하기 시작하지만 아직 체계적이지 않다. 파일명을 가끔 언급하지만 정확한 경로는 제공하지 않는다.

### Level 3 (L3.0) — Collaborator (협력자) ●

**NTRP 대응**: 3.0 (중간 속도 랠리 안정적)
**핵심 특성**: 다양한 맥락에서 AI를 독립적으로 활용한다. 구체적 요구사항을 포함한 요청을 작성한다. 대안을 물어보고 선택한다.
**판별 질문**: "AI에게 요청할 때 파일명이나 함수명을 포함하나요?" → "대부분 포함한다"
**인지 수준**: Apply (적용) — 적용하다, 실행하다, 구현하다

### Level 3.5 (L3.5) — Collaborator+ ●

**전환 중 행동**: 작업을 단계로 나누려는 시도가 보이지만 아직 분해가 거칠다. "이거 맞아?"류의 확인을 시작한다. 검증 요청이 가끔 나타나지만 체계적이지 않다.

### Level 4 (L4.0) — Pilot (조종사) ◆

**NTRP 대응**: 4.0 (포핸드, 백핸드 모두 방향+깊이 제어)
**핵심 특성**: 복잡한 문제에 AI를 통합적으로 활용한다. 작업을 단계별로 분해하여 지시한다. 결과를 비판적으로 검증한다. 제약 조건을 명시한다.
**판별 질문**: "AI가 제안한 코드를 수정 요청한 적 있나요? 이유는?" → "구체적 이유를 대며 수정 요청한다"
**인지 수준**: Analyze (분석) — 분석하다, 구별하다, 비판하다

### Level 4.5 (L4.5) — Pilot+ ◆

**전환 중 행동**: 여러 도구를 조합하여 사용하기 시작한다. 테스트와 검증을 습관적으로 요청하지만 전략적 수준까지는 아니다. AI의 한계를 경험적으로 인식하기 시작한다.

### Level 5 (L5.0) — Architect (설계자) ⬡

**NTRP 대응**: 5.0 (예측 능력 발달, 위너 생산, 고급 샷 구사)
**핵심 특성**: AI의 한계와 강점을 예측하여 태스크를 설계한다. 워크플로우를 설계하고 다른 사람을 지도한다. 전략적 판단을 내린다.
**판별 질문**: "AI가 잘 못하는 작업을 알고 있나요? 그런 경우 어떻게 대처하나요?" → "구체적 한계를 설명하고 우회 전략을 사용한다"
**인지 수준**: Evaluate (평가) — 평가하다, 비교하다, 판단하다

> **L4→L5는 "80%의 벽" 돌파 지점이다.** Addy Osmani가 말한 "AI가 80%를 만들 수 있지만 나머지 20%가 진짜 중요하다"를 체감하고, 그 20%를 주도하는 사람이 된다.

### Level 5.5 (L5.5) — Architect+ ⬡

**전환 중 행동**: 멀티에이전트 활용을 시도하기 시작한다. 자동화 파이프라인 구상이 보이지만 아직 체계화되지 않았다. 다른 사람의 AI 활용 문제를 해결해준 경험이 있다.

### Level 6 (L6.0) — Conductor (지휘자) ✦

**NTRP 대응**: 6.0 (세미프로, 전국 수준)
**핵심 특성**: AI 기반 시스템을 설계하고 운영한다. 멀티에이전트 팀을 구성하고 조율한다. 조직/팀 수준의 AI 활용 역량을 개발한다.
**판별 질문**: "AI를 활용한 반복 가능한 워크플로우나 시스템을 만든 적 있나요?" → "설계하고 운영 중이다"
**인지 수준**: Create (창조-응용) — 설계하다, 구성하다, 통합하다

### Level 6.5 (L6.5) — Conductor+ ✦

**전환 중 행동**: 자신만의 AI 활용 방법론을 정립하기 시작한다. 커뮤니티나 팀에서 AI 활용 경험을 공유한다. 도구의 한계를 극복하는 커스텀 솔루션을 개발한다.

### Level 7 (L7.0) — Forgemaster (대장장이) ◈

**NTRP 대응**: 7.0 (세계 최상위)
**핵심 특성**: AI 활용의 새로운 방법론이나 패러다임을 창조한다. 커뮤니티/산업에 기여한다. Cherny 패턴 수준의 시스템 사고를 실현한다.
**판별 질문**: "다른 사람들이 참고하는 AI 활용 방법론, 도구, 프레임워크를 만든 적 있나요?" → "만들었고, 다른 사람들이 사용하고 있다"
**인지 수준**: Create (창조-혁신) — 창조하다, 발명하다, 혁신하다

## 6축 x 7레벨 행동 기술자 매트릭스

각 칸에는 **관찰 가능한 행동**(~를 한다)과 **자동 감지 신호**를 함께 기술한다. 0.5 레벨은 전환 중 행동이다.

### DECOMP (작업 분해) — Builder의 포핸드

> 복잡한 요청을 AI가 처리 가능한 단위로 나누는 행동

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | 하나의 큰 요청을 통째로 전달한다. "이 앱 만들어줘" 수준. 단계 개념 없음. | avg_msg_len < 100자, 번호 목록 비율 = 0% |
| **L1.5** | 가끔 "먼저 이것부터"라는 표현이 등장하지만 체계적이지 않다. | avg_msg_len 100~200자, 순서 표현 비율 < 5% |
| **L2.0** | 요청에 2개 이상의 항목을 나열한다. "A도 하고 B도 해줘". 순서와 의존관계는 고려하지 않는다. | avg_msg_len 150~300자, 나열형 요청 비율 > 10% |
| **L2.5** | "먼저 X하고 그 다음 Y해줘" 형태의 순서 지정이 나타나기 시작한다. 2단계 분해. | 순서 접속사 포함 비율 > 15% |
| **L3.0** | 작업을 3단계 이상으로 명시적으로 분해한다. 각 단계에 구체적 산출물을 기대한다. | 번호 목록 포함 메시지 비율 > 20%, 단계 수 평균 3+ |
| **L3.5** | 단계 간 의존관계를 인식한다. "A가 끝나야 B를 할 수 있다"는 식의 조건부 분해. | 조건 표현 포함 비율 > 10% |
| **L4.0** | 작업을 기능/모듈 단위로 분해한다. 각 단계에 입력/출력을 명시한다. 병렬 처리 가능한 작업을 구분한다. | 파일/함수 단위 지시 비율 > 40%, 평균 분해 단계 4+ |
| **L4.5** | "스파이크 → 구현" 패턴이 나타난다. 불확실한 부분을 먼저 탐색한 후 본 작업을 진행한다. | 탐색형 요청 후 구현 요청 패턴 > 20% |
| **L5.0** | 전체 프로젝트를 위크/스프린트 단위로 설계하고 AI에게 각 단계를 위임한다. 기술적 의존성 그래프를 구성한다. | 프로젝트 구조 언급 > 30%, 다중 세션 연속 작업 패턴 |
| **L5.5** | 작업 분해 후 각 단계를 다른 에이전트/도구에 배분하기 시작한다. | has_orchestration = True, orch_tool_count > 3 |
| **L6.0** | 멀티에이전트 팀에 역할과 책임을 배분한다. 작업 분해가 시스템 아키텍처 설계 수준. | teamName 존재, orch_tool_count > 10 |
| **L6.5** | 자동 분해 템플릿이나 워크플로우를 만들어 반복 사용한다. | 커스텀 명령어/스킬 사용 |
| **L7.0** | 새로운 작업 분해 방법론이나 프레임워크를 창안하고, 다른 사람도 사용할 수 있도록 체계화한다. | 외부 공유 증거 (블로그, 오픈소스) |

### VERIFY (검증 전략) — Builder의 백핸드

> AI 출력물을 비판적으로 검토하고 품질을 확인하는 행동

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | AI 출력물을 그대로 수용한다. 검증 행동 없음. "고마워"로 대화 종료. | verification_ratio = 0, correction_ratio = 0 |
| **L1.5** | 결과가 기대와 다를 때 불만족을 표현하지만 구체적 검증은 하지 않는다. | correction_ratio < 0.05 |
| **L2.0** | "이거 맞아?"류의 확인 질문을 한다. 판단 기준은 "느낌". | verification_ratio > 0, followup_ratio > 0.03 |
| **L2.5** | 특정 부분을 지적하며 "이 부분이 이상한데"라고 말한다. 원인 분석까지는 하지 않는다. | correction_ratio > 0.05 |
| **L3.0** | 실행 결과를 확인한다. "돌려봤더니 에러가 나", "결과가 예상과 다르다"는 피드백을 준다. | verification_ratio > 0.15, 실행 결과 언급 비율 > 10% |
| **L3.5** | 테스트를 요청하기 시작한다. "테스트 코드도 만들어줘", "이 케이스도 확인해봐". | "테스트"/"test" 키워드 포함 비율 > 5% |
| **L4.0** | 체계적 검증 절차를 수행한다. 경계 조건, 엣지 케이스를 지적한다. 코드 리뷰를 요청한다. | verification_ratio > 0.2, correction_ratio > 0.1 |
| **L4.5** | AI의 답변을 다른 관점에서 재검증한다. "다른 방법으로도 확인해봐", "반례가 있을까?" | alternative_ratio > 0.05, followup_ratio > 0.1 |
| **L5.0** | 검증 체크리스트나 품질 기준을 사전에 정의한다. "이 기준을 만족하는지 확인해". 검증을 위임하고 결과만 리뷰한다. | constraint_ratio > 0.1 |
| **L5.5** | 자동화된 검증 파이프라인을 구성한다. CI/CD, 자동 테스트, 린팅을 활용한다. | bypass_ratio > 0.5, 자동화 도구 키워드 |
| **L6.0** | 검증 시스템을 설계한다. 팀 수준의 코드 리뷰 프로세스, 품질 게이트를 구축한다. | 시스템 수준 검증 키워드 |
| **L6.5** | 검증 결과를 메트릭으로 추적하고 품질 트렌드를 분석한다. | 메트릭 키워드, 추적/분석 패턴 |
| **L7.0** | 새로운 검증 방법론이나 품질 프레임워크를 창안한다. AI 출력물 평가의 새 기준을 제시한다. | 외부 공유 증거 |

### ORCH (오케스트레이션) — Builder의 발리

> 도구, 에이전트, 워크플로우를 조합하여 활용하는 행동

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | 단일 도구의 기본 기능만 사용한다. 채팅에서 텍스트 대화만. | tool_diversity <= 2 |
| **L1.5** | 도구의 다른 기능(파일 첨부 등)을 가끔 사용한다. | tool_diversity = 2~3 |
| **L2.0** | 2~3개 도구를 순차적으로 사용한다. 도구 간 연결은 수동(복사-붙여넣기). | tool_diversity = 3~4 |
| **L2.5** | 파일 읽기, 검색 등 내장 기능을 활용하기 시작한다. | tool_diversity = 4 |
| **L3.0** | 4~6개 도구를 목적에 맞게 선택하여 사용한다. Read + Edit + Bash 기본 체인. | tool_diversity = 4~6, tools_per_turn > 3 |
| **L3.5** | 도구 선택 이유를 인식한다. "이건 Bash로 하는 게 나을 것 같아". | 도구 지시 키워드, tool_diversity > 5 |
| **L4.0** | 7~9개 도구를 조합하여 복잡한 워크플로우를 구성한다. 도구 간 데이터 흐름을 설계한다. | tool_diversity = 7~9, tools_per_turn > 5 |
| **L4.5** | 자주 쓰는 도구 조합을 패턴화한다. "Grep으로 찾고 Read로 읽고 Edit으로 수정" 자동화. | 반복 도구 패턴, tool_diversity > 8 |
| **L5.0** | 멀티에이전트를 활용하기 시작한다. Task/Agent 도구로 작업을 위임한다. | has_orchestration = True, orch_tool_count > 5 |
| **L5.5** | 에이전트 간 역할 분담을 설계한다. 병렬 처리와 순차 처리를 구분한다. | orch_tool_count > 8 |
| **L6.0** | 팀 기반 멀티에이전트 시스템을 운영한다. 작업 분배, 결과 통합을 주도한다. | teamName 존재, orch_tool_count > 15 |
| **L6.5** | 커스텀 MCP 서버, CI/CD 파이프라인을 통합한 워크플로우를 구축한다. | mcp_tool_count > 5, bypass_ratio > 0.7 |
| **L7.0** | 새로운 오케스트레이션 패턴이나 프레임워크를 창안한다. 도구 생태계에 기여한다. | tool_diversity > 15, 외부 기여 증거 |

### FAIL (실패 대응) — Builder의 풋워크

> 빌드 에러, 런타임 오류, 예상치 못한 결과에 대처하는 행동

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | 오류 발생 시 "고쳐줘"만 요청한다. 에러 메시지를 읽지 않는다. 같은 요청을 반복한다. | 동일 메시지 반복 비율 > 30% |
| **L1.5** | 에러 메시지를 복사하여 전달하기 시작한다. 해석은 AI에게 전적으로 맡긴다. | 에러 메시지 붙여넣기 패턴 |
| **L2.0** | 에러 메시지를 읽고 "이 에러가 뭐야?"라고 물어본다. 의미를 이해하려 시도한다. | "에러"/"error" + "뭐야"/"무슨" 패턴 |
| **L2.5** | 에러의 위치(파일, 줄 번호)를 인식하고 언급한다. "X파일 Y줄에서 에러가 난다". | 파일 경로 + 줄 번호 동시 언급 |
| **L3.0** | 에러 원인을 대략적으로 추측한다. "아마 X 때문인 것 같아"라고 가설을 제시한다. | 원인 추측 패턴 > 10% |
| **L3.5** | 에러 해결 후 "왜 이 에러가 났는지"를 물어본다. 재발 방지를 의식한다. | 원인 분석 질문 비율 > 5% |
| **L4.0** | 에러 원인을 분석하고 두 가지 이상의 해결 방안을 고려한다. "A 방법과 B 방법 중 뭐가 나아?" | alternative_ratio > 0.05 |
| **L4.5** | 에러를 예방하는 방향으로 요청한다. "이런 에러가 안 나도록 처음부터 설계해줘". | 예방 키워드("미리", "방지", "prevent") |
| **L5.0** | 에러 패턴을 인식하고 체계적으로 대응한다. 유형별 대응 전략이 있다. | recovery 성공률 > 70% |
| **L5.5** | 에러 대응을 자동화한다. 린팅, 타입 체크, 사전 검증으로 에러를 사전에 차단한다. | 자동 검증 도구 사용 |
| **L6.0** | 팀/시스템 수준의 에러 대응 프로세스를 설계한다. 롤백 전략을 구축한다. | 시스템 수준 에러 대응 키워드 |
| **L6.5** | 에러 데이터를 수집하고 분석하여 시스템을 개선한다. 사후 분석(post-mortem)을 수행한다. | 분석/메트릭 추적 패턴 |
| **L7.0** | 새로운 에러 대응 방법론이나 탄력성 패턴을 창안하고 공유한다. | 외부 기여 증거 |

### CTX (맥락 관리) — Builder의 서브

> AI에게 파일 경로, 기술 스택, 제약 조건 등 배경 정보를 제공하는 행동

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | 맥락 정보 없이 요청한다. "이거 해줘". 배경, 목적, 제약이 전혀 없다. | context_specificity_ratio < 0.3 |
| **L1.5** | 가끔 목적을 언급한다. "~하려고 하는데"라는 표현이 나타나기 시작한다. | 목적 표현 비율 < 10% |
| **L2.0** | 요청에 배경 설명을 추가한다. "지금 X 프로젝트를 하고 있는데, Y가 필요해". | context_specificity_ratio > 0.4 |
| **L2.5** | 파일명이나 기술 스택을 언급하기 시작한다. "React로 만들고 있어", "index.js에서". | context_specificity_ratio > 0.5 |
| **L3.0** | 정확한 파일 경로, 함수명, 변수명을 제공한다. 코드 스니펫을 첨부한다. | context_specificity_ratio > 0.6, 코드 블록 포함 > 15% |
| **L3.5** | 제약 조건을 명시하기 시작한다. "기존 코드는 건드리지 마", "TypeScript로". | constraint_ratio > 0.03 |
| **L4.0** | 기능적/비기능적 요구사항을 구분하여 제공한다. 성능, 보안, 호환성 조건을 명시한다. | constraint_ratio > 0.05 |
| **L4.5** | 프로젝트 아키텍처와 설계 의도를 설명한다. "이 모듈은 X 역할이고 Y와 연결된다". | 아키텍처 키워드, avg_msg_len > 600 |
| **L5.0** | 컨텍스트 창 관리를 의식적으로 수행한다. CLAUDE.md로 맥락을 사전 주입한다. | 맥락 파일 참조 패턴 |
| **L5.5** | 맥락 제공을 자동화한다. 프로젝트 설정 파일, 컨텍스트 템플릿을 사전 준비한다. | 설정 파일 참조, 템플릿 사용 |
| **L6.0** | 팀/프로젝트 수준의 맥락 관리 체계를 구축한다. CLAUDE.md/AGENTS.md 시스템을 설계한다. | 구조화된 맥락 파일 다수 |
| **L6.5** | 맥락의 품질을 지속적으로 개선한다. 어떤 맥락이 AI 성능에 영향을 미치는지 분석한다. | 맥락 최적화/A/B 비교 |
| **L7.0** | 새로운 맥락 관리 방법론이나 프레임워크를 창안하고 공유한다. | 외부 기여 증거 |

### META (메타인지) — Builder의 게임 전략

> 자신의 AI 활용 패턴을 인식하고 전략적으로 조정하는 행동. 전체 세션의 5.7%에서만 관찰되는 가장 희귀한 차원.

| 레벨 | 행동 기술자 | 자동 감지 신호 |
|------|-----------|--------------|
| **L1.0** | AI 활용 방식에 대한 성찰 없음. "그냥 써본다". | metacognitive_ratio = 0 |
| **L1.5** | "AI가 잘 해주네" / "AI가 못 해주네"라는 수준의 인상적 평가. | 평가 표현 있으나 구체적 기준 없음 |
| **L2.0** | AI가 잘하는 것과 못하는 것을 경험적으로 느끼기 시작한다. | 경험적 평가 패턴 (비체계적) |
| **L2.5** | 요청 방식이 결과에 영향을 미친다는 것을 인식한다. "이렇게 물어보니까 더 잘 해주네". | 요청 방식 언급 패턴 |
| **L3.0** | 다른 사람의 프롬프트 기법을 참고하여 적용한다. 역할 부여 등 기법을 시도한다. | 프롬프트 기법 키워드 |
| **L3.5** | 어떤 기법이 어떤 상황에 효과적인지 경험적으로 구분하기 시작한다. | 상황별 기법 선택 패턴 |
| **L4.0** | AI 활용의 장단점을 분석적으로 논의한다. "이 접근의 트레이드오프는...". | strategic_ratio > 0.05, metacognitive_ratio > 0 |
| **L4.5** | 자신의 AI 활용 패턴을 의식적으로 개선하려 한다. "이전에는 이렇게 했는데 이번엔 저렇게 해볼게". | 자기 참조 패턴 |
| **L5.0** | AI 활용 전략을 사전에 수립한다. "이 프로젝트는 이런 방식으로 AI를 활용하겠다". | strategic_ratio > 0.1 |
| **L5.5** | 자신의 AI 활용 데이터를 분석한다. 어떤 패턴이 효과적이었는지 정량적으로 파악한다. | 데이터 분석 패턴 |
| **L6.0** | AI 활용 방법론을 체계화하고 문서화한다. 팀에 교육한다. | 방법론 키워드, 교육/공유 패턴 |
| **L6.5** | 다양한 AI 활용 전략을 비교 실험하고 최적의 접근법을 도출한다. | A/B 비교, 실험 키워드 |
| **L7.0** | AI 활용의 새로운 패러다임이나 사고 프레임워크를 창안하고 산업에 기여한다. | 외부 기여 증거 |

## 레벨 판정 공식

### Fit Score 계산

각 세션에서 레벨 L에 대한 적합도 점수(Fit Score)를 계산한다:

```
F_L = DECOMP_score × 0.25
    + VERIFY_score × 0.25
    + ORCH_score   × 0.15
    + FAIL_score   × 0.15
    + CTX_score    × 0.10
    + META_score   × 0.10

각 차원 score = 해당 레벨 행동 기술자의 자동 감지 신호 충족 비율 (0.0~1.0)
```

### 최종 레벨 결정

```
1. 각 차원에서 정수 레벨과 0.5 레벨 기준 충족도 계산
2. 가중 평균으로 raw_score 산출 (소수점 2자리)
3. 0.5 단위로 반올림하여 공식 레벨 결정

반올림 기준:
  x.00 ~ x.24 → x.0
  x.25 ~ x.74 → x.5
  x.75 ~ x.99 → (x+1).0
```

### 게이트 조건

Fit Score가 아무리 높아도 게이트를 통과하지 못하면 해당 레벨로 판정되지 않는다.

| 게이트 | 필수 조건 | 근거 |
|--------|----------|------|
| L3 진입 | context_specificity_ratio > 0.5 | 파일/함수 수준의 구체성이 없으면 L3 미만 |
| L4 진입 | verification_ratio > 0.15 AND correction_ratio > 0.05 | Builder는 검증 없이 L4 불가 |
| L5 진입 | (tool_diversity > 8 OR has_orchestration) AND strategic_ratio > 0.05 | 도구 다양성과 전략적 사고 필수 |
| L6 진입 | teamName 존재 OR orch_tool_count > 10 | 멀티에이전트 운용 경험 필수 |
| L7 진입 | L6 게이트 + 외부 기여 증거 | 산업/커뮤니티 기여 필수 |

### 3회 일관성 원칙

일회성 성과는 인정하지 않는다. NTRP가 "일관되게 할 수 있는 것"을 기준으로 레벨을 판정하듯:

```
승급 조건:
  최근 5개 세션 중 3개 이상에서 상위 레벨 Fit Score가 현 레벨보다 높아야 함
```

## 승급/강등 기준

### NTRP 비대칭 원칙

심리적 안전과 동기 부여를 위해, 승급은 빠르게, 강등은 느리게 설계한다.

| 방향 | 조건 | 사용자 경험 |
|------|------|-----------|
| **승급** | 2회 연속 상위 레벨 Fit Score → **즉시 승급** | 즉각적 보상, 성취감 |
| **강등** | 3회 연속 하위 레벨 Fit Score + **사용자 동의** | 동의 없이 강등하지 않음 |

### 80% 경계선 (L4→L5) 특별 취급

L4→L5는 "80%의 벽"을 돌파하는 핵심 순간이다. Addy Osmani가 지적한 "AI가 80%를 만들 수 있지만 나머지 20%가 진짜 중요하다"를 체감하고 넘어서는 지점.

- **승급 조건 강화**: L4→L5는 3회 연속 (일반 승급은 2회)
- **승급 시 특별 이벤트**: "벽 돌파" 메시지와 함께 성장 증거를 상세히 표시
- **의미**: "만드는 사람"(Pilot)에서 "설계하는 사람"(Architect)으로의 전환

### Strike 시스템 (과소평가 방지)

자기평가보다 실측이 높으면 자동 상향한다:

```
if 동적_레이팅 > 자기평가_레벨 + 1.0 (3회 연속):
    Strike 발생 → 3 Strikes면 자동 승급
```

### 바닥 효과 보정

- 첫 세션 완료 즉시 최소 L1.5 부여 (AI를 사용했다는 사실 자체가 L1 초과)
- 3세션 이내에 기본 도구 2종 이상 사용 시 L2.0으로 자동 승급
- L1은 "AI를 처음 접하는 순간"에만 해당

### 정체기 대응

같은 레벨에 4주 이상 머무르면 마이크로 마일스톤을 제공한다:

```
예시 (L3 → L4 정체 시):
  [ ] 검증 요청 비율 20% 달성
```
